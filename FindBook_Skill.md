# FindBook Skill：書籍重點整理 SOP

## 使用時先詢問

請先問使用者：

1. 這次要找哪些主題？
2. 每個主題各需要幾本新書？
3. 要搜尋哪一段日期區間？
4. 是否要用 ChatGPT 與 Gemini 可用的最高多工上限，同步整理多本書籍？

日期區間格式：

- 開始日期：`YYYY-MM-DD`
- 結束日期：`YYYY-MM-DD`
- 若使用者沒有指定，預設找最近 30 天內的新書、暢銷書或熱門書。

主題固定為：

- `01_商業創業`
- `02_心理勵志`
- `03_自然科學`
- `04_醫療保健`
- `05_飲食養生`
- `06_電腦資訊`
- `07_其他`

## 找資料

1. 到博客來、灰熊愛讀書、momo、讀冊等網站，找日期區間內的新書、暢銷書、熱門書、推薦書或評分較高的書。
2. 若網站提供出版日期、上架日期、榜單日期或文章發布日期，必須優先用該日期判斷是否落在日期區間內。
3. 若網站沒有明確日期，仍可列入候選，但 `sourceDateNote` 要標註「來源未提供明確日期」。
4. 每筆至少記錄：書名、作者、來源網站、來源網址、榜單名稱、擷取日期、日期區間、來源日期說明。
5. 將新取得的「書名 + 作者」與 `data.json` 的既有索引比對。
6. 若已存在，略過。
7. 若不存在，進入分類與整理流程。

## 分類規則

1. 每本書只能歸到 7 大主題中的 1 類。
2. 若可跨類，選最主要閱讀目的作主題，其餘放入 `tags`。
3. 投資、創業、職涯、管理優先歸 `01_商業創業`。
4. 習慣、情緒、人際、自我成長優先歸 `02_心理勵志`。
5. 物理、宇宙、生物、數學、科普優先歸 `03_自然科學`。
6. 疾病、醫療、心理健康、照護優先歸 `04_醫療保健`。
7. 食譜、營養、減脂、代謝、養生優先歸 `05_飲食養生`。
8. AI、程式、資料、演算法、軟體工具優先歸 `06_電腦資訊`。
9. 歷史、文化、文學、生活雜學或不適合前六類者歸 `07_其他`。

## 多工整理規則

1. 先將所有待整理新書建立成工作佇列，每個工作至少包含：書名、作者、分類、來源資訊、`searchDateRange`。
2. ChatGPT 與 Gemini 各自使用獨立 worker pool，不互相等待；同一本書可同時啟動 ChatGPT 重點與 Gemini 重點整理。
3. 每次執行前先確認當下 ChatGPT 與 Gemini 可穩定處理的多工上限，分別記為 `chatgptMaxWorkers` 與 `geminiMaxWorkers`。
4. 多工上限以實際服務、帳號、瀏覽器分頁、MCP 操作穩定性與速率限制為準；不得繞過服務限制、建立額外帳號或用會觸發封鎖的方式提高併發。
5. 若無法事先確認上限，從 1 個 worker 開始，逐步增加到 2、3、4...；一旦出現限流、回覆品質下降、分頁錯亂或操作不穩，就回退到上一個穩定值。
6. ChatGPT worker 數不得超過 `chatgptMaxWorkers`；Gemini worker 數不得超過 `geminiMaxWorkers`。
7. 每個 worker 一次只處理一本書的一種來源重點，完成並驗證 100 點後才領取下一個工作。
8. 必須用「書名 + 作者 + 來源模型」追蹤每個工作，避免 ChatGPT 與 Gemini 結果寫入錯誤書籍。
9. 若某個 worker 失敗，將該工作標記為待重試，不得阻塞其他 worker 繼續處理。
10. 所有可同步處理的書籍都應依目前穩定上限同時推進；只有遇到服務限制、品質問題或使用者要求時才降為單工。

## 網頁版來源要求

1. `chatgptHighlights` 必須來自網頁版 ChatGPT 的實際最終回覆；`geminiHighlights` 必須來自網頁版 Gemini 的實際最終回覆。
2. 不得用 Codex 自行生成、本地模板、快取舊結果、API 回覆、摘要改寫或未經網頁驗證的文字替代網頁版 ChatGPT/Gemini 的重點整理。
3. 每本書都必須在對應模型的網頁版送出提示詞，等待該網頁完成回覆，再擷取該次網頁對話的最後一則 assistant/model 回覆。
4. 寫入 JSON 的重點內容必須和網頁版最後驗收通過的回覆逐行一致；只允許移除前後空白、拆成 JSON array，不得改寫正文。
5. 若回覆格式不合格，必須回到同一個網頁版對話要求 ChatGPT 或 Gemini 修正；修正後寫入的版本也必須是網頁版實際回覆。
6. 若無法確認已送出到網頁版、無法確認模型、無法取得完整回覆，或無法證明內容來自網頁版，不得標記為 `complete`。
7. 未完成網頁版驗證時，該模型狀態必須標記為 `pending_web_verification`，且不得宣稱已取得網頁版 ChatGPT/Gemini 重點整理。

## ChatGPT 重點

1. 使用 Codex 可用的瀏覽器/MCP 操作使用者已登入的網頁版 ChatGPT，不得用 Codex 本身的回答替代 ChatGPT 網頁版輸出。
2. 進入 ChatGPT 頁面，並啟用 ChatGPT 最高階 pro 的 model；多工時依 `chatgptMaxWorkers` 開啟可穩定操作的獨立分頁或對話。
3. ChatGPT 提示詞只能放「正確輸出格式」與「驗收規則」，不要在提示詞中放錯誤範例，避免模型模仿錯誤字樣。
4. ChatGPT 提示詞需要求：書籍 100 個重點整理，每個重點格式必須是 `01、重點正文` 到 `100、重點正文`，以繁體中文文字輸出；除了這 100 行重點，其餘文字都不要。
5. `NN、` 後面必須直接接重點正文，不得加筆記類型、閱讀角度、分類標籤、小標題、前言、結語、模型自述或固定尾句。
6. 每一點不得使用 `｜`；不得使用「短標籤加冒號」作為開頭；不得輸出任何不是重點正文的欄位名稱或格式說明。
7. 取得 ChatGPT 網頁版回覆後，檢查是否剛好 100 點。
8. 若不足、超過或編號錯誤，必須在 ChatGPT 網頁版同一對話要求補齊、刪減或重排到剛好 100 點。
9. 將 ChatGPT 網頁版最後驗收通過的結果寫回 `index.html` 會讀取的相關 JSON 中，存入該書的 `chatgptHighlights`，並將 `chatgptStatus` 設為 `complete`。
10. 多本書同時整理時，依 `chatgptMaxWorkers` 平行處理；每本書使用獨立提示與獨立結果暫存，避免內容交叉污染。

## Gemini 重點
1. 使用 Codex 的 MCP 連接並操作使用者已登入 Gemini 的 Chrome。
2. 進入 Gemini 頁面，並啟用Gemini 最高階pro的model；多工時依 `geminiMaxWorkers` 開啟可穩定操作的獨立分頁或對話。
3. 將該書的「書名 + 作者」丟給 Gemini，要求產出該書籍的 100 個重點整理。
4. Gemini 提示詞只能放「正確輸出格式」與「驗收規則」，不要在提示詞中放錯誤範例，避免模型模仿錯誤字樣。
5. Gemini 提示詞需要求：書籍的 100 個重點整理，格式為 `01、重點正文` 到 `100、重點正文`，以繁體中文文字輸出；除了這 100 行重點，其餘文字都不要。
6. `NN、` 後面必須直接接重點正文，不得加筆記類型、閱讀角度、分類標籤、小標題、前言、結語、模型自述或固定尾句。
7. 每一點不得使用 `｜`；不得使用「短標籤加冒號」作為開頭；不得輸出任何不是重點正文的欄位名稱或格式說明。
8. 取得 Gemini 回覆後，檢查是否剛好 100 點。
9. 若不足、超過或編號錯誤，要求 Gemini 補齊、刪減或重排到剛好 100 點。
10. 將 Gemini 網頁版最後驗收通過的結果寫回 `index.html` 會讀取的相關 JSON 中，存入該書的 `geminiHighlights`，並將 `geminiStatus` 設為 `complete`。
11. 確認同一本書同時保留 `chatgptHighlights` 與 `geminiHighlights`，讓 `index.html` 可以同時顯示 ChatGPT 和 Gemini 的重點整理。

## 重點格式驗收

1. ChatGPT 與 Gemini 的輸出都必須先通過格式驗收，通過後才可寫入 JSON。
2. 必須剛好 100 行，且編號必須依序為 `01、` 到 `100、`。
3. 每行只允許 `NN、重點正文`；`NN、` 後若出現筆記類型、閱讀角度、分類標籤、小標題、固定尾句或格式說明，視為不合格。
4. 任一行包含 `｜`，或在正文開頭使用短標籤加冒號，視為不合格。
5. 任一行不是完整重點句、只是分類名稱、只是提示詞欄位，或包含模型自述、前言、結語，視為不合格。
6. 驗收只檢查格式與來源，不得由 Codex 改寫重點正文。
7. 若輸出不合格，只能回到對應網頁版模型要求重寫不合格的行或整份 100 點；不得把不合格文字直接寫入 JSON。

## 寫入 JSON

1. 更新 `data.json` 的 `books` 索引。
2. 依分類寫入對應 JSON。
3. 每本書格式：

```json
{
  "id": "book-id",
  "title": "書名",
  "author": "作者",
  "sourceName": "來源榜單",
  "sourceUrl": "https://example.com",
  "sourceDateNote": "出版日期、上架日期、榜單日期或來源未提供明確日期",
  "searchDateRange": {
    "from": "YYYY-MM-DD",
    "to": "YYYY-MM-DD"
  },
  "tags": ["標籤"],
  "summary": "短摘要",
  "updatedAt": "YYYY-MM-DD",
  "chatgptHighlights": ["01、..."],
  "chatgptStatus": "complete",
  "geminiHighlights": ["01、..."],
  "geminiStatus": "complete",
  "highlightsSource": "web_chatgpt_and_web_gemini",
  "highlightsCapturedAt": "YYYY-MM-DDTHH:mm:ss+08:00"
}
```

## 驗證

1. 所有 JSON 必須能被 JSON parser 讀取。
2. `data.json` 中的書籍索引不得重複同一組「書名 + 作者」。
3. 每本新書的 ChatGPT 及 Gemini 的重點需為 100 點。
4. `chatgptStatus: complete` 只能用於已確認來自網頁版 ChatGPT 的內容；`geminiStatus: complete` 只能用於已確認來自網頁版 Gemini 的內容。
5. 抽查每本書的第 1、50、100 點，必須和對應網頁版最後回覆一致。
6. 若 ChatGPT 或 Gemini 已補齊，補齊後的重點也需為 100 點，且必須來自同一個網頁版服務。
7. 新書來源必須保留日期區間與來源日期說明。
8. 開啟 `index.html` 後，主題下拉選單能顯示新書，書名下拉選單能切換資料。
9. 不可有亂碼文字。
